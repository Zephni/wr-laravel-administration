Updating & Version Handling

WRLA ships with a version handling system that upgrades your application in a controlled, repeatable way. Rather than just pulling new package code, each release that needs migration work defines its own version update — a single class that performs whatever steps are required: running composer, mutating config files, asking questions, and so on.

1 How updates are applied

The currently applied version is stored in public/vendor/wr-laravel-administration/version.json. When an update runs, the VersionHandler discovers every version update definition, sorts them in ascending order, and runs only those newer than your stored version — updating the stored version after each one succeeds.

Versions use a three-digit-per-group scheme, e.g. 0.1.001, 0.1.002, …, 0.1.999, 0.2.000.

2 Running an update

After updating the package, run the update command from your terminal: 

php artisan wrla:update

Developers (users for whom developer tools are enabled) can also run updates from the admin panel via the Dev Tools → Update WRLA modal — see Running from the modal below for how the output is displayed.

3 Composer behaviour (--no-dev)

Version updates that run composer update decide whether to pass --no-dev based on the current app environment. Configure which environments should exclude dev dependencies under the developer.composer.no_dev config key: 

'developer' => [
    'enable' => fn($wrlaUserData) => false,

    'composer' => [
        // App environments that should run `composer update --no-dev`
        'no_dev' => ['production'],
    ],
],

4 Writing a version update

Each version update is its own file in src/Classes/VersionHandler/Versions/, named after the version (e.g. Version_0_1_001.php). It extends VersionUpdate and implements three methods: 

// src/Classes/VersionHandler/Versions/Version_0_1_001.php

class Version_0_1_001 extends VersionUpdate
{
    public function version(): string
    {
        return '0.1.001';
    }

    public function title(): string
    {
        return 'Composer update & developer config restructure';
    }

    public function run(VersionUpdateContext $context): void
    {
        // Run composer update (honours developer.composer.no_dev)
        if (! $this->runComposerUpdate($context)) {
            throw new \RuntimeException('composer update failed.');
        }

        // ...then apply any custom changes, e.g. migrate a config key
        $context->info('Migrating configuration...');
    }
}
Throwing an exception from run() aborts the update — the version is not marked as applied, so it will be retried on the next run.

5 The update context

Updates can be triggered from the console (interactive) or the web dev-tools modal (buffered). To behave identically in both, a version update must route all output, questions and shell commands through the VersionUpdateContext it is given — never call echo or read STDIN directly.

Method Purpose
line() / info() / warn() / error() Write output at the relevant severity
ask($key, $question, $default) Ask a free-text question
confirm($key, $question, $default) Ask a yes/no question
choice($key, $question, $choices, $default) Ask the user to pick from a list
runProcess($command) Run a shell command, streaming/capturing its output
change($status, $message) Report whether a step changed, made no change, partially changed, or was skipped
Console vs web: in the console, questions are fully interactive. In the web modal, questions resolve to their supplied answer or default (Livewire cannot block mid-request). Prefer driving decisions from config — as with developer.composer.no_dev — so updates behave predictably in both contexts.

6 Reporting what changed

Rather than guessing from the raw output, a version update can explicitly tell the user whether each step actually changed something, made no change, only partially changed, or was skipped. Use $context->change($status, $message) with a VersionChangeStatus case:

Status Meaning
VersionChangeStatus::Changed The step applied a change successfully
VersionChangeStatus::Unchanged Nothing needed doing — already in the desired state
VersionChangeStatus::Partial Only some of the intended change could be applied
VersionChangeStatus::Skipped The step could not run (eg. a required file was missing) 
use WebRegulate\LaravelAdministration\Classes\VersionHandler\VersionChangeStatus;

if (! file_exists($configPath)) {
    $context->change(VersionChangeStatus::Skipped, 'Config not found, migration skipped.');
    return;
}

if ($alreadyMigrated) {
    $context->change(VersionChangeStatus::Unchanged, 'Config already present, nothing to do.');
    return;
}

file_put_contents($configPath, $updated);
$context->change(VersionChangeStatus::Changed, 'Config migrated to the new structure.');

Each call is written to the output at a sensible severity and recorded, so once the version finishes the handler automatically prints a concise summary, e.g. Summary of changes - Changed: 1, Unchanged: 2.

Future proof: need a new outcome? Add a case to VersionChangeStatus with its label, prefix and severity — the context output and the summary pick it up automatically, no other changes required.

7 Running from the modal

How the web Update WRLA modal executes updates is controlled by the developer.update.mode config key: 

'developer' => [
    // ...

    'update' => [
        // 'live'     - run in the background and stream the output as it happens
        // 'blocking' - run synchronously and show the full output once finished
        'mode' => 'live',
    ],
],
Mode Behaviour
live (default) Spawns wrla:update as a detached background process that writes to a log file; the modal polls and streams the output live. If the host cannot spawn a background process, it automatically falls back to blocking.
blocking Runs the update synchronously within the request and shows the full output once it finishes. The most portable option if background processes are restricted on your host.
