Site Configuration

An optional key/value store for app-wide settings, managed from the WRLA admin panel and readable anywhere in your application.

What is it?

Site Configuration gives you a database-backed, admin-managed settings store without the overhead of writing your own. Each entry is a simple key / type / value row you can read, write, and soft-delete from both the admin panel and your application code.

  • Manage settings from the WRLA admin panel — no code deploys needed for value changes.
  • Five value types: text, json, comma_delimited_array, boolean, base64.
  • Values are strongly typed when read back — JSON returns an array, booleans return a real bool, etc.
  • Per-request in-memory cache so repeated lookups don't hammer the database.
  • Soft-delete support — entries can be restored from the admin panel.

Installation

Run the dedicated artisan command. It is interactive and will walk you through each step: 

php artisan wrla:site-configuration

The command performs the following steps:

1 Creates the migration

Sets up the database table that will store all your configuration entries. You'll get a preview of the migration before being asked whether to run it, so there are no surprises.

2 Publishes the model

Adds the SiteConfiguration Eloquent model to your project so you can read and write configuration values from anywhere in your application. The model is yours to extend as needed.

3 Publishes the manageable model

Adds the SiteConfiguration ManageableModel so you can manage your site's configuration from within the admin panel. This file is fully customisable — feel free to adjust the fields, validation, and layout to suit your needs.

4 Registers the navigation item

Adds a Site Configuration link to your admin panel navigation so your team can find and manage settings straight away. The link stays hidden until the migration has been run, so it's safe to add it at any point during setup.

Note: Every step is idempotent. Re-running the command after a partial install is safe — already-existing files and nav items are detected and skipped (with your confirmation).

Using in the Admin Panel

Once installed, a Site Configuration item will appear in the WRLA navigation (below the Dashboard). From here you can:

  • Browse all configuration entries, filtered by key or soft-delete status.
  • Create a new entry — provide a unique key, choose a type, enter a value, and optionally add a description.
  • Edit any entry inline.
  • Soft-delete entries and restore them later without data loss.

The type field controls how the stored value is decoded when read back via code. Choose the type that matches your intended data shape:

Type Stored as Returned as
textPlain stringstring
jsonJSON stringarray (associative)
comma_delimited_arrayComma-separated stringarray of trimmed strings
booleanTruthy/falsy stringbool
base64Base64 encoded stringDecoded string (or null)

Using in Your Application

All interaction goes through the published App\Models\SiteConfiguration model. The most common methods are shown below.

Reading a value

Use getValue() for a quick typed read. Pass an optional default that is returned when the key does not exist. 

use App\Models\SiteConfiguration;

// Returns a string (or null if the key doesn't exist)
$siteName = SiteConfiguration::getValue('site_name');

// Returns a bool with a safe fallback
$maintenanceMode = SiteConfiguration::getValue('maintenance_mode', false);

// Returns an array when the type is json or comma_delimited_array
$allowedDomains = SiteConfiguration::getValue('allowed_domains', []);

Getting the full model

When you need more than just the value (e.g. the description or type), use getByKey(). 

$config = SiteConfiguration::getByKey('site_name');

if ($config) {
    echo $config->key;         // 'site_name'
    echo $config->value;       // raw stored string
    echo $config->description; // optional admin-facing note
    echo $config->resolveValue(); // strongly-typed value
}

Writing / updating a value

Use setValueByKey() to create or update an entry from code. It handles soft-deleted duplicates automatically — a previously deleted row for the same key will be restored and updated rather than creating a new one. 

use App\Models\SiteConfiguration;
use WebRegulate\LaravelAdministration\Enums\SiteConfigurationType;

// Create or update a plain text entry
SiteConfiguration::setValueByKey(
    key: 'site_name',
    value: 'My Application',
    type: SiteConfigurationType::Text,
    description: 'The public-facing name of the site.'
);

// Create a boolean flag
SiteConfiguration::setValueByKey(
    key: 'maintenance_mode',
    value: 'false',
    type: SiteConfigurationType::Boolean,
);
Caching: Lookups are cached in memory for the lifetime of the request. If you update a value and need to read it back in the same request, the cache is automatically invalidated via model events — so getValue() will always return the freshest value.
